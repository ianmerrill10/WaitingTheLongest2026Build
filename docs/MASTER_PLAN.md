# WaitingTheLongest.com — Master Plan

## Mission

Surface every shelter dog in America and rank them by how long they've been waiting for a home. The longest waiters get the most visibility.

## The Problem

Millions of dogs enter shelters every year. Many get overlooked — especially older dogs, less "photogenic" breeds, and dogs in rural shelters with no online presence. The longer a dog waits, the more invisible it becomes.

## The Solution

WaitingTheLongest.com flips the script: **the longer a dog waits, the MORE visible it becomes.** Dogs are ranked by `intake_age_days` (days since intake), with the longest-waiting dogs featured prominently.

Instead of scraping data from third parties (legally risky, technically fragile), we provide a **free Shelter Intake API** that shelters use to submit their own dogs. This creates a verified, accurate, real-time database.

## Data Strategy: Shelter Intake API

The API spec is defined in `openapi.yaml` (designed by Opus 4.8, SAC-aligned). Key design decisions:

### Shelter Onboarding
1. Shelter applies at POST /v1/organizations (public, no auth needed)
2. Gets a Tier 0 (self-asserted) API key immediately — never blocked
3. Auto-promoted to Tier 1 via EIN match (IRS Publication 78), .gov domain, or aggregator confirmation
4. Tier 2 (trusted) for established partners with track record

### Data Submission
- Single animal: POST /v1/animals (idempotent upsert on org + external_id)
- Bulk: POST /v1/animals/batch (up to 1000 records, async processing)
- Status updates: POST /v1/animals/{id}/status (cheapest write — mark adopted, pending, etc.)
- Photos: Presigned upload URLs, EXIF stripped, content screened, perceptual hash for dedup

### Data Quality
- SAC-aligned vocabularies (species, intake_type, outcome_type) — see docs/SAC_VOCABULARY_MAPPING.md
- Per-record confidence scoring (validation completeness + submitter tier + cross-source corroboration)
- Deduplication via canonical_id merging
- Provenance tracking (submitted_at, last_refreshed_at, last_verified_at)
- RFC 9457 error responses with field-level detail

### Why This Beats Scraping
1. **Legal:** No TOS violations, no cease-and-desists
2. **Accuracy:** Shelters update their own data in real-time
3. **Trust:** Users know every listing is from a verified source
4. **Scalability:** Adding shelters = handing out API keys, not writing scrapers
5. **Industry standard:** SAC vocabulary alignment = credibility with major welfare orgs

## Architecture

Two APIs:

### 1. Shelter Intake API (Private, authenticated)
Defined in `openapi.yaml`. Shelters write data here.
- Auth: X-API-Key header (per-org)
- Base: api.waitingthelongest.com

### 2. Public Read API (Public, no auth)
Frontend consumes this. Needs to be built:
- GET /v1/public/dogs — browse/search/filter, sorted by days_waiting desc
- GET /v1/public/dogs/{id} — dog detail with photos, shelter info
- GET /v1/public/shelters/{id} — shelter profile + their dogs
- GET /v1/public/stats — platform stats (total dogs, avg wait, adoptions today)
- GET /v1/public/featured — longest-waiting dogs, daily highlights

- **LIGHT MODE** for all UI — no dark themes
- **No fake/mock data** — use zeros or "N/A" if real data unavailable
